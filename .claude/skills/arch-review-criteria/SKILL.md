# 아키텍처 리뷰 기준

architecture-reviewer agent와 /arch-review command가 참조하는 평가 기준 및 출력 형식.

## 평가 기준

### 1. SOLID 원칙

- **단일 책임 원칙 (SRP)**: 클래스/메서드가 하나의 책임만 가지는가? Service가 Controller 역할을 겸하지 않는가?
- **개방-폐쇄 원칙 (OCP)**: 새 기능 추가 시 기존 코드 수정 없이 확장 가능한가?
- **의존성 역전 원칙 (DIP)**: 구체 클래스가 아닌 인터페이스에 의존하는가? `new` 직접 생성 남용 없는가?

### 2. 추상화 수준

- **과도한 추상화**: 한 번만 쓰이는 인터페이스, 불필요한 Factory, 과도한 레이어링
- **부족한 추상화**: 코드 중복 (3회 이상 반복 로직), 인라인 복잡 로직
- **잘못된 추상화 경계**: Service가 Repository 로직 직접 포함, Controller가 비즈니스 로직 포함

### 3. 복잡도

- **함수 길이**: 50줄 초과 시 분리 검토
- **중첩 깊이**: 4레벨 초과 (if-for-if-for) 시 추출 필요
- **불필요한 복잡도**: 과도한 설정 옵션, 이해하기 어려운 흐름
- **인지 복잡도**: 읽는 사람이 흐름을 따라가기 위해 얼마나 많은 컨텍스트를 기억해야 하는가?

### 4. 테스트 용이성 (설계 관점)

- 모킹하기 어려운 의존성: `static` 메서드, 전역 상태, 생성자 내 `new` 직접 생성
- 부작용이 많은 함수: 외부 I/O + 상태 변이가 한 메서드에 혼재
- ※ 커버리지 갭("무엇을 테스트해야 하는가")은 qa-reviewer 담당

### 5. 프로젝트 일관성

- 기존 패턴과의 불일치: 다른 서비스에서 쓰는 패턴과 다른가?
- 백엔드 변경 시: `docs/guides/backend.md` 가이드와 일치하는가?
- 프론트엔드 변경 시: `docs/guides/frontend.md` 가이드와 일치하는가?
- 패키지 구조: `config/controller/service/repository/domain/dto/util` 규칙 준수?
- UniPlan 도메인 레이어: 외부 요청 Headers(`X-User-Id`, `X-User-Email`) 사용 패턴 일치?

## 출력 형식

```
## 요약
[변경사항 한 줄 요약]

## 발견된 문제
1. [심각도: Critical/High/Medium/Low] [문제]: [설명] → [개선 방안]

## 권장 사항
- [선택적 개선사항]

## 결론
[승인 / 수정 필요 / 재작업 필요]
```

## 심각도 기준

| 심각도 | 판단 기준 | 조치 |
|--------|----------|------|
| Critical | SOLID 위반으로 확장 불가, 테스트 불가능한 구조 | 수정 필수 (Phase 3 재실행) |
| High | 복잡도 초과, 일관성 위반 | 강력 권장 수정 |
| Medium | 개선 가능한 설계 | 선택적 수정 |
| Low | 스타일/네이밍 | 참고만 |
