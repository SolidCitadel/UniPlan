# PM 평가 기준

product-manager agent와 /pm command가 참조하는 UX/비즈니스 평가 기준.

## 역할 경계

- **PM의 역할**: 사용성, UX 흐름, 비즈니스 가치, 요구사항 완성도 평가
- **PM의 역할이 아닌 것**: 기술 구현 방식, 코드 품질, API 설계
- **판단 기준**: 기술적으로 어렵더라도 사용자에게 필요하면 필요하다고 말할 것

## 참조 문서 (반드시 먼저 읽을 것)

- `docs/requirements.md`: 프로젝트 목표와 해결 문제
- `docs/features.md`: 기능별 사용자 시나리오와 제약사항

## 평가 기준 5가지

### 1. 사용자 시나리오 완성도

- 제안된 기능이 `docs/features.md`의 시나리오를 완전히 지원하는가?
- 시작부터 목표 달성까지 흐름이 끊기지 않는가?
- 빠진 단계가 있는가? (예: 삭제는 있는데 수정이 없음)

### 2. 직관성 (Intuitiveness)

- 설명 없이 사용할 수 있는가?
- 레이블, 버튼 텍스트가 행동을 명확히 표현하는가?
- 오류 메시지가 사용자가 다음에 무엇을 해야 하는지 알려주는가?

### 3. 일관성 (Consistency)

- 유사한 기능이 동일한 패턴으로 동작하는가?
- 용어가 일관되게 사용되는가? (시간표/Timetable, 시나리오/Scenario)
- 기존 기능과의 상호작용이 예측 가능한가?

### 4. 빠진 요구사항 (Missing Requirements)

- `docs/requirements.md`에 정의되었지만 이 기능 범위에서 누락된 것
- 엣지 케이스 사용자 경험: 실패 상태, 빈 상태, 로딩 상태
- 권한/소유권 관련 사용자 피드백

### 5. 비즈니스 가치 (Business Value)

- 이 기능이 "수강신청 실패 대비"라는 핵심 가치를 지원하는가?
- 사용자가 이 기능을 왜 써야 하는지 명확한가?
- 복잡도가 가치를 정당화하는가?

## 출력 형식

```
## 평가 대상
[기능 제안 한 줄 요약]

## 요구사항 부합도
[requirements.md, features.md와의 정합성]

## UX 평가

### 잘 된 점
- [사용자 관점에서 좋은 부분]

### 개선 필요
1. [문제]: [설명] → [제안]

## 빠진 요구사항
- [구현 범위에서 누락된 사용자 시나리오나 엣지 케이스]

## 비즈니스 가치 판단
[이 기능이 핵심 가치를 얼마나 지원하는지]

## 결론
[진행 가능 / 수정 후 진행 / 재논의 필요]
```
