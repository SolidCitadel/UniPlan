# ADR-001: 모노레포 구조 선택

**상태**: 승인됨
**작성일**: 2026-02-24

## 배경 (Context)

다수의 백엔드 서비스와 프론트엔드가 존재하는 구조에서, 소스 코드 저장소 전략 결정이 필요함. 1인 개발 환경이며, 프론트엔드-백엔드 간 API 계약 변경이 빈번하게 발생하는 상황. 또한 AI 코딩 도구(Claude Code 등)를 적극 활용하는 개발 방식을 채택하고 있음.

## 고려한 대안 (Alternatives Considered)

| 대안 | 장점 | 단점 |
|------|------|------|
| **모노레포 (선택)** | 원자적 커밋, 일관된 CI/CD, 의존성 파악 용이 | 레포 크기 증가, 서비스별 독립 버전 관리 어려움 |
| 멀티레포 (서비스별) | 서비스별 독립 배포/버전 관리, 팀별 코드 소유권 명확 | 크로스 서비스 변경 시 다수 PR 필요, 통합 테스트 구성 복잡 |

## 결정 (Decision)

**모노레포** 구조를 채택함. 프론트엔드, 백엔드 전체 서비스, 통합 테스트, 인프라 설정을 단일 저장소에서 관리함.

### 디렉터리 구조

```
UniPlan/
├── app/
│   ├── backend/           # Spring Boot MSA (Gradle 멀티모듈)
│   │   ├── api-gateway/
│   │   ├── user-service/
│   │   ├── catalog-service/
│   │   └── planner-service/
│   └── frontend/          # Next.js
├── tests/integration/     # 전체 서비스 대상 통합 테스트
├── docker-compose.yml     # 전체 인프라 정의
└── docs/                  # 공유 문서
```

## 근거 (Rationale)

- **원자적 변경**: API 계약 변경 시 백엔드 DTO, 프론트엔드 타입, 통합 테스트를 하나의 커밋으로 반영 가능. 멀티레포에서는 서비스별로 별도 PR을 생성하고 배포 순서를 조율해야 함
- **통합 테스트 용이**: `docker-compose.yml`과 `tests/integration/`이 같은 레포에 있어 전체 시스템 테스트 구성이 단순함
- **1인 개발 환경**: 서비스별 독립 배포 주기, 팀별 코드 소유권 분리, 코드 접근 제어 등 멀티레포의 장점은 다수 팀 운영 시 의미가 있으며, 이 프로젝트에서는 해당 없음
- **AI 도구 활용 최적화**: AI 코딩 도구가 프로젝트 전체 맥락(백엔드 API 계약, 프론트엔드 타입, 테스트 코드, 인프라 설정)을 단일 저장소에서 파악할 수 있어, 크로스 서비스 변경 시 정확도와 생산성이 높아짐

## 영향 (Consequences)

### 긍정적 영향
- 프론트-백 API 계약 불일치를 커밋 단위로 방지 가능
- 공유 Docker Compose 설정으로 전체 환경을 한 곳에서 관리
- 프로젝트 전체 구조를 한눈에 파악 가능

### 부정적 영향 / 트레이드오프
- 레포 크기 증가에 따른 클론/CI 시간 증가 (현재 규모에서는 무시 가능)
- 서비스별 독립적인 릴리스 태깅이 복잡해질 수 있음
- 팀 규모 확장 시 멀티레포로의 전환 필요성 발생 가능
