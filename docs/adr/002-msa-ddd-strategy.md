# ADR-002: MSA 및 도메인 주도 설계 (MSA & DDD Strategy)

**상태**: 승인됨  
**작성일**: 2026-01-20

## 배경 (Context)

수강신청 시스템은 성격이 다른 도메인들이 공존함:
- **강의 카탈로그**: Read-heavy, 학기 단위로 거의 불변, 향후 검색 고도화(Elasticsearch 등) 가능성
- **시간표/시나리오 계획**: Write-heavy, 사용자별 개인 데이터, 복잡한 비즈니스 로직
- **사용자 인증**: 보안 민감, 변경 빈도 낮음

모놀리식 아키텍처에서는 이들이 하나의 배포 단위에 묶여 도메인 간 결합도가 높아지고, 특정 기능의 부하가 전체 시스템에 전파됨.

## 고려한 대안 (Alternatives Considered)

| 대안 | 장점 | 단점 |
|------|------|------|
| 모놀리식 | 단순함, 트랜잭션 용이 | 스케일링 어려움, 높은 결합도 |
| **MSA + DDD (선택)** | 독립 배포/확장, 장애 격리 | 분산 시스템 복잡성 |
| 모듈형 모놀리스 | 중간 복잡도 | 완전한 독립 배포 불가 |

## 결정 (Decision)

**마이크로서비스 아키텍처(MSA)**를 채택하고, 각 서비스를 **DDD(Domain-Driven Design)** 원칙에 따라 분리합니다.

### 서비스 경계 (Bounded Contexts)

| 서비스 | 책임 | 특성 |
|--------|------|------|
| **User Service** | 사용자 인증, 프로필, 대학 정보 | 계정 보안 |
| **Catalog Service** | 강의 데이터 관리, 검색 | Read Heavy |
| **Planner Service** | 시간표, 시나리오, 수강신청 | Write Heavy, 핵심 비즈니스 |

## 근거 (Rationale)

### 기술적 근거
- **트래픽 패턴 차이**: Catalog(Read-heavy)과 Planner(Write-heavy)의 부하 특성이 상이하여, 독립적인 스케일링 전략이 합리적임
- **장애 격리**: 강의 검색 폭주 시에도 시간표 계획 기능은 영향 없음. 수강신청 시즌에 카탈로그 조회 부하가 급증하는 것이 예상되는 도메인 특성상, 격리의 실익이 존재함
- **도메인 경계 명확성**: 사용자 인증, 강의 데이터, 수강 계획은 라이프사이클과 변경 빈도가 다름. DDD Bounded Context로 자연스럽게 분리됨
- **기술 스택 유연성**: 향후 Catalog에 Elasticsearch 도입, Planner에 Redis 캐싱 도입 등 서비스별 독립적 기술 선택이 가능

### 규모에 대한 인식
- 1인 개발/단일 호스트 규모에서 MSA가 과할 수 있다는 점은 인지함. 모듈형 모놀리스로도 도메인 분리는 가능하나, 서비스 간 네트워크 통신, API 계약 설계, 독립 배포 등 **실제 분산 환경에서만 발생하는 문제**를 경험하기 위해 MSA를 선택함

## 영향 (Consequences)

### 긍정적 영향
- 서비스별 명확한 도메인 경계로 코드 이해도 향상
- 독립적인 배포 및 확장 가능
- 팀 규모 확장 시 병렬 개발 용이

### 부정적 영향 / 트레이드오프
- 분산 데이터 관리의 복잡성 (Service 간 조인 불가, Eventual Consistency)
- 인프라 복잡도 증가 (Gateway, Service Discovery 등)
- 통합 테스트 비용 증가
